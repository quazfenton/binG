# Codebase Review: Boundaries & Security Policies

## Overview
The Boundaries and Security Policies layer provides the protective "jail" for agentic execution. It ensures that agents—whether running in a cloud sandbox, local CLI, or desktop app—cannot perform destructive operations outside their authorized workspace or execute dangerous system-level commands.

## Key Components

### 1. Workspace Boundary Utilities (`workspace-boundary.ts`)
The primary path validation engine.
- **VFS Awareness**: Automatically recognizes `/project/` and `/workspace/` prefixes as "safe" internal paths that never require boundary checks.
- **Normalization Security**: Implements a custom `normalizePath` that collapses `..` segments *without* relying on Node's `path` module, ensuring consistency between browser, CLI, and Rust (desktop) contexts.
- **Destructive Operation Filtering**: Maintains a strict set of `DESTRUCTIVE_OPERATIONS` (write, delete, move, etc.) that trigger confirmation logic if they target paths outside the resolved workspace root.

### 2. Sandbox Security Manager (`security-manager.ts`)
Provides deep "Defense in Depth" for cloud execution environments.
- **Cross-Platform Normalization**: A sophisticated engine that detects and fixes Windows paths (e.g., `C:/foo`) accidentally passed to Linux-based sandboxes (E2B/Daytona). This prevents silent path resolution failures.
- **Dangerous Keyword Blocking**: Blocks 20+ dangerous command partials (e.g., `sudo`, `curl | sh`, `chmod 777`) to prevent privilege escalation or data exfiltration.
- **Metacharacter Protection**: Explicitly blocks shell metacharacters like `` ` ``, `$`, and `\n` to prevent command injection via tool arguments.

### 3. Desktop Security Policy (`security.ts`)
A configurable policy engine for local execution.
- **Risk Assessment**: Categorizes commands into `low`, `medium`, `high`, and `critical` risk levels based on regex patterns.
- **Approval Gates**: Defines `approvalRequiredPatterns` (e.g., `rm -rf`, `sudo`) that the UI uses to pause execution and prompt the user for explicit consent.
- **Audit Logging**: Maintains a rolling buffer of the last 1000 commands for security forensics and user accountability.

## Findings

### 1. Exceptional Path Normalization
The `resolvePath` method in `security-manager.ts` is highly resilient. Its ability to strip Windows drive letters and handle embedded paths (e.g., `/foo/C:/bar`) addresses a common "blind spot" in cross-platform agentic systems where paths generated by an LLM on Windows are executed in a Linux container.

### 2. Comprehensive Command Blocking
The `BLOCKED_COMMAND_PATTERNS` in `security.ts` are extremely thorough, covering everything from fork bombs to bind mounts and Python/PHP data exfiltration servers. This indicates a high level of security maturity.

### 3. Consistency Across Tiers
The sharing of the `WorkspaceBoundaryResult` interface across CLI, Desktop, and Web ensures that the "confirmation" UX is consistent regardless of how the user is interacting with the platform.

## Logic Trace: Running a Dangerous Command
1.  **Agent** attempts to execute `sudo rm -rf /`.
2.  **Sandbox Security Manager** intercepting the `exec_shell` call.
3.  **Command Sanitization**: `sudo` is detected in `DANGEROUS_COMMAND_PARTIALS`.
4.  **Error Thrown**: "Security Exception: Command blocked by security policy".
5.  **Agent Gateway**: Catches the error and returns a sanitized "Blocked" message to the LLM.
6.  **Audit Log**: The attempt is recorded in the session's security audit trail.

## Recommended Actions

| Action | Priority | Reason |
| :--- | :--- | :--- |
| **Centralize Normalization** | High | The path normalization logic is slightly different between `workspace-boundary.ts` and `security-manager.ts`. Unify these into a single shared utility to prevent "edge-case" drift. |
| **Strict Metacharacter Blocking** | Medium | The `sanitizeCommand` method in `security-manager.ts` is currently `@deprecated` in favor of `validateCommand`, but it still holds stronger metacharacter checks. Port these checks to the main validation engine. |
| **Honeypot Files** | Low | Add "Honeypot" files (e.g., `/.aws/credentials.fake`) to the sandbox image to detect if an LLM is actively attempting to exfiltrate sensitive data. |
| **Dynamic Policy Updates** | Low | Allow users to "allowlist" specific risky commands (like `npm install -g`) for their session without disabling the entire security manager. |
