# V1 vs V2 Architecture - Implementation Guide

This document describes the architecture patterns for connecting binG to various agentic engines, covering local CLI binaries, remote HTTP services, and containerized implementations.

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         binG Engine Selector                           │
├─────────────────────────────────────────────────────────────────────────────┤
│  V1 (API)              │  V2-CLI            │  V2-HTTP-SDK   │ Docker │
│  ─────────            │  ─────────         │  ───────────   │ ────── │
│  OpenAI API           │  Local spawn      │  Remote HTTP   │ E2B    │
│  Anthropic API        │  --server --port  │  SDK calls    │ Daytona │
│  Any OpenAI-compat    │  stdio-jsonrpc    │  REST API    │ Sprites │
│                       │                  │              │        │ WebContainer│
└─────────────────────────────────────────────────────────────────────────────┘
```

## Architecture Types

| Type | Description | Communication | Model Passing |
|------|-------------|---------------|--------------|
| `v1-api` | LLM API calls (OpenAI, Anthropic) | HTTP REST | `model` in request body |
| `v2-cli` | Local binary spawn via `spawnLocalAgent()` | stdin/stdout JSON-RPC or CLI args | CLI arg `--model` or env var |
| `v2-http-sdk` | Remote HTTP service | HTTP SDK/REST | `model` in request body or header |
| `v2-container` | Containerized (E2B, Daytona, etc.) | Mounted filesystem + exec | Env var (`OPENCODE_MODEL`) |

### OpenCode Classes

#### 1. OpencodeProvider (`opencode-spawn.ts`)

Basic CLI spawn with JSON over stdin/stdout:

```typescript
// Pattern: spawn → stdin write → stdout parse → tool call loop
class OpencodeProvider implements LLMProvider {
  async runAgentLoop(options: LLMAgentOptions): Promise<LLMAgentResult> {
    const binaryPath = findOpencodeBinarySync() ?? 'opencode';
    const proc = spawn(binaryPath, ['chat', '--json'], { env: {...} });

    // Send prompt via stdin
    proc.stdin.write(JSON.stringify({ prompt, tools: [...] }) + '\n');

    // Parse JSON-Lines from stdout
    proc.stdout.on('data', (chunk) => {
      const parsed = JSON.parse(line);
      if (parsed.tool_call) {
        // Execute tool, write result back
        proc.stdin.write(JSON.stringify({ tool_result: {...} }) + '\n');
      }
    });
  }
}
```

**CLI args:** `chat --json [--model MODEL]`
**Model:** Via `--model` CLI arg or `OPENCODE_MODEL` env var
**Event protocol:** JSON-Lines over stdout

#### 2. OpencodeEventStream (`opencode-event-stream.ts`)

# Design Principles

This section captures architectural design guidance for the V1/V2 system.

## Core Principle: v1 vs v2 is Architectural, Not Capability-Based

```
v1 = app-native orchestration layer    (app controls context, memory, tools)
v2 = external autonomous runtime     (engine controls its own state)
```

This distinction is about **control ownership**, not quality.

## The Runtime Contract

Every agent runtime (local CLI, remote HTTP, container) should implement:

```typescript
interface AgentRuntime {
  // Lifecycle
  startSession(config?: SessionConfig): Promise<Session>;
  sendPrompt(prompt: string): AsyncIterable<RuntimeEvent>;
  interrupt(): Promise<void>;
  resume(): Promise<void>;
  endSession(): Promise<void>;

  // State
  getState(): Promise<RuntimeState>;
  forkSession(): Promise<Session>;

  // Configuration
  switchModel(model: string): Promise<void>;
  updateSystemPrompt(prompt: string): Promise<void>;
}

// Normalized event stream
type RuntimeEvent =
  | { type: 'token'; text: string }
  | { type: 'tool-call'; tool: string; args: Record<string, any> }
  | { type: 'tool-result'; tool: string; result: ToolResult }
  | { type: 'file-edit'; path: string; diff: string }
  | { type: 'plan-update'; plan: string[] }
  | { type: 'error'; error: string }
  | { type: 'status'; status: 'running' | 'waiting' | 'complete' }
  | { type: 'complete'; result: string };
```

## Best Practices

### 1. Prefer Persistent Runtimes

**WRONG:**
```text
spawn → run → exit
```

**RIGHT:**
```text
persistent server/runtime
    ↓
attach sessions
    ↓
stream events
```

Use `opencode serve` or `opencode web` to start a persistent server, then attach to it. This preserves memory and eliminates cold starts.

### 2. Model Selection Hierarchy

**BEST:** Per-request override
```bash
opencode run --model claude-3-5-sonnet "prompt"
```

**SECOND:** Session-scoped model
```bash
opencode session create --model claude-opus
```

**WORST:** Restarting infrastructure

### 3. Normalize Events, Not Raw Output

Every runtime adapter must convert to the normalized `RuntimeEvent` format. Never pipe raw CLI output directly to UI.

### 4. Runtime Pools for Scale

```typescript
class RuntimePool<T> {
  idle: AgentRuntime[] = [];
  busy: Set<AgentRuntime> = new Set();
  factory: () => T | Promise<T>;

  async acquire(): Promise<T> {
    if (this.idle.length > 0) {
      return this.idle.pop()!;
    }
    const runtime = await this.factory();
    this.busy.add(runtime);
    return runtime;
  }

  release(runtime: T): void {
    this.busy.delete(runtime);
    this.idle.push(runtime);
  }
}
```

### 5. Shadow Memory for State Synchronization

Because v2 runtimes have internal memory hidden from the orchestrator, maintain external summaries:

```typescript
type ShadowMemory = {
  sessionId: string;
  summaries: string[];
  importantFiles: string[];
  toolCalls: ToolCall[];
  lastPlan: string[];
};

async syncShadowMemory(runtime: AgentRuntime): Promise<ShadowMemory> {
  // Poll runtime state and build external summary
  // Enables: migrations, model switches, fork/recover
}
```

### 6. Agent Forking for Exploration

```typescript
// Instead of one agent doing everything
mainAgent
  ├── branch: ui-exploration
  ├── branch: backend-optimization
  └── branch: testing
       ↓
       merge best result

async forkSession(sessionId: string): Promise<string> {
  // Creates copy of session state for parallel exploration
}
```

### 7. Execution Feedback Loops

```typescript
async autonomousRepairLoop(prompt: string, maxRetries = 3): Promise<string> {
  for (let i = 0; i < maxRetries; i++) {
    const result = await runtime.sendPrompt(prompt);

    // Compile
    const compileResult = await compileWorkspace();

    // Runtime test
    const runtimeResult = await runTests();

    if (compileResult.success && runtimeResult.success) {
      return result;
    }

    // Feed back failure context
    prompt += `\n\nError: ${compileResult.error || runtimeResult.error}\nFix this.`;
  }
}
```

This is the critical difference between an "AI assistant" and an "autonomous engineering system."
